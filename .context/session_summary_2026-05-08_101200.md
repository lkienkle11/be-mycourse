## Overview

Follow-up to sub15: after registering the WebP decoder (previous fix), the CREATE and GET responses still returned `width_bytes: 0, height_bytes: 0`. Root cause: `ToMediaEntity` calls `BuildTypedMetadata` with `payload=nil`, so `buildImageTypedMetadata` calls `ImageSizeFromPayload(nil)` → `(0,0)`, ignoring the persisted `width`/`height` already stored in the DB's `MetadataJSON`. Fixed by adding a fallback to `raw["width"]`/`raw["height"]` in `buildImageTypedMetadata`, and removed debug print statements added during investigation.

## Key decisions

- **Fallback pattern mirrors video**: `buildVideoTypedMetadata` already reads `raw["width"]`/`raw["height"]` from persisted metadata; image now does the same when payload-decode returns 0.
- **No double-decode at read time**: Correct design — DB rows don't carry payload bytes; dimensions come from persisted JSON, not live re-decode.
